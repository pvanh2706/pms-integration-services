## Naming Conventions

### Projects

| Pattern | Example |
|---|---|
| `PmsIntegration.<Layer>` | `PmsIntegration.Application` |
| `PmsIntegration.Providers.<Name>` | `PmsIntegration.Providers.Tiger` |
| `PmsIntegration.<Layer>.Tests` | `PmsIntegration.Providers.Fake.Tests` |

### Classes

| Type | Convention | Example |
|---|---|---|
| Interface | `I` prefix + noun | `IPmsProvider`, `IQueuePublisher` |
| Use-case handler | `<Action>Handler` | `ReceivePmsEventHandler` |
| Options POCO | `<Name>Options` | `TigerOptions`, `PmsSecurityOptions` |
| DI extension class | `<Name>ServiceExtensions` | `TigerServiceExtensions` |
| Mapper | `<Provider>Mapper` | `TigerMapper` |
| Request builder | `<Provider>RequestBuilder` | `TigerRequestBuilder` |
| HTTP client wrapper | `<Provider>Client` | `TigerClient` |
| Provider class | `<Provider>Provider` | `TigerProvider` |
| Background service | describes the behavior | `ProviderConsumerOrchestrator` |

### Files

- One public type per file.
- File name matches type name, e.g. `TigerProvider.cs`.
- No generic `Helper`, `Util`, or `Manager` suffixes.

### Methods

- Async methods end in `Async`, e.g. `HandleAsync`, `PublishAsync`, `SendAsync`.
- Boolean methods use `Is`, `Has`, `Can`, or `Try` prefixes, e.g. `TryAcquire`.

## ProviderCode Rules

A **ProviderCode** (also called `ProviderKey`) is the canonical identifier for a provider throughout the entire system: config, queues, message headers, and DI.

### Format

- **Uppercase ASCII only**, e.g. `TIGER`, `OPERA`, `FAKE`
- Words separated by underscores for multi-word names, e.g. `MY_PROVIDER`
- No spaces, no hyphens, no lowercase

### Normalization

Before any comparison or lookup, apply:

```csharp
providerCode.Trim().ToUpperInvariant()
```

`IPmsProviderFactory.Get(providerCode)` performs this normalization internally, so callers do not need to normalize before calling `Get`.

### Consistency rule

The ProviderCode must be **identical** in all of the following locations:

| Location | Example |
|---|---|
| `IPmsProvider.ProviderKey` property | `"TIGER"` |
| `appsettings.json` ? `Providers:<KEY>` | `"Providers:TIGER"` |
| `appsettings.json` ? `Queues:ProviderQueues:<KEY>` | `"Queues:ProviderQueues:TIGER"` |
| DI extension config section | `configuration.GetSection("Providers:TIGER")` |
| Named `HttpClient` identifier | `"TIGER"` |
| RabbitMQ message header `providerKey` | `"TIGER"` |
| `IntegrationJob.ProviderKey` field | `"TIGER"` |

A mismatch between any two of these will cause a runtime `InvalidOperationException` from `PmsProviderFactory.Get`.

## Queue Naming

| Queue type | Pattern | Example (Tiger) |
|---|---|---|
| Main | `q.pms.<providerKeyLower>` | `q.pms.tiger` |
| Retry | `<main>.retry` | `q.pms.tiger.retry` |
| DLQ | `<main>.dlq` | `q.pms.tiger.dlq` |

Rules:
- Provider key in the queue name is **lowercase**.
- All three queues are declared as durable.
- Queue names are never hard-coded; they come from `Queues:ProviderQueues:<KEY>` in `appsettings.json`.
- `RabbitMqTopology` derives the retry and DLQ names by appending `.retry` / `.dlq` to the main queue name.

## Correlation IDs

A **correlation ID** is a GUID string that traces a single PMS event through the entire pipeline.

### Lifecycle

| Stage | Action |
|---|---|
| PMS sends event | `PmsEventEnvelope.CorrelationId` is set by the PMS (or generated by `ReceivePmsEventHandler` if absent) |
| HTTP response | `X-Correlation-Id` response header is set by `PmsEventController` |
| `IntegrationJob` | `IntegrationJob.CorrelationId` is copied from the envelope |
| RabbitMQ message | `correlationId` header on the AMQP message |
| Provider HTTP call | `X-Correlation-Id` request header set by each provider's `RequestBuilder` in `ProviderRequest.Headers`; `CorrelationIdHandler` (in `Infrastructure/Http/DelegatingHandlers/`) exists for this purpose but must be wired to each named `HttpClient` explicitly via `.AddHttpMessageHandler<CorrelationIdHandler>()` |
| All log entries | Always include `correlationId` as a structured field |

### Consumer rule

When a message is moved to the `.retry` or `.dlq` queue, the `correlationId` header **must be preserved unchanged**. Never strip or overwrite correlation IDs.

## Logging Fields

All structured log entries must include these fields. Serilog enriches them automatically where possible; use `LogContext.PushProperty` for the rest.

### Required fields (all log levels)

| Field | Type | Source |
|---|---|---|
| `correlationId` | `string` (GUID) | `IntegrationJob.CorrelationId` |
| `hotelId` | `string` | `IntegrationJob.HotelId` |
| `eventId` | `string` | `IntegrationJob.EventId` |
| `eventType` | `string` | `IntegrationJob.EventType` |
| `providerKey` | `string` | `IntegrationJob.ProviderKey` |

### Optional fields (include when available)

| Field | Type | When |
|---|---|---|
| `jobId` | `string` (GUID) | All consumer/handler logs |
| `attempt` | `int` | Retry logs |
| `x-last-error-code` | `string` | On failure |
| `x-last-error-message` | `string` | On failure |

### Audit actions

Fixed audit action strings logged by `ElasticAuditLogger`:

| Action | When |
|---|---|
| `pms.received` | Event accepted by `PmsEventController` |
| `job.enqueued` | Job published to the main queue |
| `job.processing` | Consumer starts processing a job |
| `job.success` | Provider API call succeeded |
| `job.retryable_failed` | Provider API call failed with a retryable outcome (5xx, 408, 429, network error) |
| `job.failed` | Provider API call failed with a **non-retryable** outcome (4xx except 408/429, mapping error) |
| `job.provider_not_registered` | `IPmsProviderFactory.Get` could not find a provider for the key |
| `job.duplicate_ignored` | Idempotency check rejected the job as a duplicate |

> **Note:** Moving a job to the DLQ is logged via `ILogger` (`LogWarning`) inside `ProviderConsumerService`, not via `IAuditLogger`. There is currently no `job.dlq` audit action emitted.

Audit log format:

```
AUDIT {Action} {@Data}
```

> **Note on field name:** `ElasticAuditLogger` uses the template parameter `{Action}`. Serilog serializes this as the structured property `Action` (PascalCase). Kibana queries in this documentation reference `auditAction:` — verify the actual field name in your Elasticsearch index once the Elasticsearch sink is wired and rename the template parameter to `{auditAction}` if needed for consistency.

Example:

```json
{
  "level": "Information",
  "message": "AUDIT job.success",
  "correlationId": "3fa85f64-...",
  "hotelId": "H001",
  "eventId": "EVT-001",
  "eventType": "Checkin",
  "providerKey": "TIGER",
  "jobId": "7c9e6679-...",
  "attempt": 1
}
```

### Elasticsearch / Kibana

- Serilog is bootstrapped by `SerilogElasticSetup.Configure` in `Infrastructure/Logging/`.
- **Current state:** only the **Console** sink is active. The `WriteTo.Elasticsearch(...)` call is provided as a commented-out production template inside `SerilogElasticSetup.cs`. To enable it, add `Serilog.Sinks.Elasticsearch` and replace the comment block.
- Kibana version: **6.8.23**.
- All logs go to a **single index** (when Elasticsearch is wired). Do not create per-provider indices; use the `providerKey` field to filter.

## Folder Layout Rules

### Host

```
Host/
  Controllers/      ? API endpoint controllers only
  Middleware/       ? Cross-cutting request pipeline concerns
  Background/       ? IHostedService implementations (queue consumers)
  Options/          ? POCO option classes bound in Program.cs
  Providers/        ? ProvidersServiceExtensions.cs (single composition point)
  Program.cs
```

**Forbidden in Host:** provider-specific mapping, business logic, RabbitMQ connection details.

### Core

```
Core/
  Abstractions/     ? Interfaces only
  Contracts/        ? Data shapes (POCOs, no logic)
  Domain/           ? Domain value types (e.g. IntegrationOutcome)
```

**Forbidden in Core:** any NuGet package reference to RabbitMQ, Serilog, HttpClient, EF, Redis.

### Application

```
Application/
  UseCases/         ? ReceivePmsEventHandler, ProcessIntegrationJobHandler
  Services/         ? EventValidator, ProviderRouter, RetryClassifier
  DI/               ? ApplicationServiceExtensions.cs
```

**Forbidden in Application:** direct RabbitMQ or HttpClient usage; Infrastructure type references.

### Infrastructure

```
Infrastructure/
  RabbitMq/         ? Connection, topology, publisher, headers
  Logging/          ? ElasticAuditLogger, SerilogElasticSetup
  Idempotency/      ? InMemoryIdempotencyStore, RedisIdempotencyStore
  Http/
    DelegatingHandlers/  ? CorrelationIdHandler
  Config/           ? AppSettingsConfigProvider
  Clock/            ? SystemClock
  Options/          ? RabbitMqOptions, QueueOptions
  Providers/        ? PmsProviderFactory
  DI/               ? InfrastructureServiceExtensions.cs
```

### Provider

```
Providers.<Name>/
  <Name>Options.cs
  <Name>RequestBuilder.cs
  <Name>Client.cs
  Mapping/
    <Name>Mapper.cs
  DI/
    <Name>ServiceExtensions.cs
```

## Forbidden References

The table below lists `ProjectReference` combinations that are **never allowed**.

| From | May NOT reference | Reason |
|---|---|---|
| `Core` | Any other project in the solution | Core must have zero internal deps |
| `Core` | `RabbitMQ.Client`, `Serilog`, `HttpClient` | Infrastructure-free |
| `Application` | `Infrastructure` | Business layer must not know impl details |
| `Application` | Any `Providers.*` | Application is provider-agnostic |
| `Providers.*` | Any other `Providers.*` | Providers must be isolated |
| `Providers.*` | `Infrastructure` | Providers must not know about queues |
| `Host` | `Core` (business logic usage) | Host is composition root, not a business layer |

### Allowed reference summary

```
Host        ? Application, Infrastructure, Providers.* (DI only)
Application ? Core
Infrastructure ? Core, Providers.Abstractions
Providers.* ? Providers.Abstractions, Core
Providers.Abstractions ? Core
```
